Getting Started

Set up BrowserBleed and run your first engagement end to end.

Authorized use only. BrowserBleed is a red team tool. Only deploy against systems and accounts you are explicitly authorized to test.

Prerequisites

Required

  • Python 3.10+ — for the payload source and sessiontest.py. Check with python3 --version.
  • AWS CLI with SSO — for deploying the server. Configured in deploy/config and ~/.aws/config.
  • Go 1.21+ — to cross-compile the report server binary. Only needed if building the server from source.

For Windows payload builds

  • PowerShell 5+ (built into Windows)
  • The build script installs PyInstaller automatically into a virtualenv — no manual setup needed

For macOS / Linux builds

  • Bash and Python 3 (usually pre-installed)
  • Linux builds require sudo (PyInstaller needs root for some bootloader steps)

Optional

  • Playwright — for sessiontest.py --browser mode: pip install playwright && playwright install chromium

End-to-end workflow

  1. Deploy the report server
    The Go report server receives exfil data and serves the web UI. It runs on your EC2 instance behind nginx + HTTPS.
    # Log in to AWS SSO (only needed when session is expired)
aws sso login

# Cross-compile and push to EC2
bash deploy/deploy-binary.sh
    The deploy script pushes a 60-second temp SSH key via EC2 Instance Connect — no PEM file needed.
  2. Build a payload
    Build scripts read your server URL and API key from deploy/config automatically. Pick a preset (the app you want to disguise the payload as) and run the build for your target's OS.
    # Windows (run from repo root in PowerShell)
.\build_windows.ps1 -Preset chrome

# macOS (run on a Mac)
./build_mac.sh --preset chrome

# Linux
sudo ./build_linux.sh --preset chrome
    The built binary lands in payloads/. Add -Upload / --upload to push it to the payload server immediately. See the Build guide for all options.
  3. Deliver the payload
    The easiest delivery method is a calendar invite with the smart link embedded. When the target clicks the meeting link, their OS is detected and the matching payload downloads automatically.
    Go to Payloads → Calendar Invite, pick a preset, fill in the recipient and time, then either download the ICS and send it manually, or click Send Invite (requires Gmail or Outlook sign-in). See the Delivery guide for full details.
  4. Target runs the payload
    When the target executes the binary, BrowserBleed extracts credentials from browser profile directories, session storage, and running browser processes, then POSTs the encrypted report to your server. The payload exits cleanly with no visible side effects.
    You can watch the server logs for the incoming upload, or just refresh Reports.
  5. Review the report
    Reports appear on the Reports page with hostname, run time, and hit count. Click View to read the full output, or download as txt / csv.
    Reports are stored encrypted on disk with AES-256-GCM and auto-expire after the configured TTL (default 24h).
  6. Validate credentials
    Download results.csv from a report, then run:
    python3 sessiontest.py results.csv
    sessiontest.py tests each captured token and cookie session against its live service and reports identity and access level. See the Session Test guide for all options.

Server configuration

All key settings live in deploy/config:

DOMAIN=reports.yourdomain.com
BB_API_KEY=your-secret-key
ENCRYPTION_KEY=64-char-hex-key   # openssl rand -hex 32
REGION=us-east-1
INSTANCE_ID=i-xxxxxxxxxxxxxxxxx

The BB_API_KEY is used to authenticate the web UI (entered at /login), the payload upload endpoint, and the build agent. Keep it secret.

The ENCRYPTION_KEY encrypts all stored reports at rest. If you lose this key, existing reports cannot be decrypted.
